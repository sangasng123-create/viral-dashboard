## 1. 이 대시보드가 하는 일 (한 줄 요약)

구글 스프레드시트의 **완료시트(누구한테 얼마 줬나 = 비용)** 와 **성과DB(NT로 들어온 유입·결제 = 매출)** 를
Python에서 자동으로 짝지어(매칭) **플랫폼·작업자별 효율(ROAS)** 을 보여주는 Streamlit 대시보드.

```
구글 스프레드시트                Python (data_loader.py)              화면 (app.py)
 ├ 완료시트(비용)   ──CSV 다운로드──▶  비용 ↔ 매출 매칭/집계  ──▶  KPI · 차트 · 작업리스트
 └ 성과DB(매출)                        (nt 키 기준)
```

- **자동 갱신**: 5분마다 시트 최신값을 다시 읽음 (사이드바 "원본 다시 불러오기"로 즉시 갱신 가능)
- **별도 DB·서버 없음**: 시트가 곧 데이터베이스. 시트만 잘 관리하면 됨

## 2. ⭐ 꼭 지켜야 할 것 (가장 중요 — 이거 안 지키면 숫자가 안 맞음)

대시보드 정확도의 **90%는 코드가 아니라 "시트에 데이터를 어떻게 적느냐"** 에 달려 있다.
완료시트 안내문에 이미 적힌 4가지 규칙을 반드시 지킬 것:

1. **NT 파라미터를 글 링크에 꼭 붙이기** ← 제일 중요
   - NT 파라미터가 없으면 성과DB에 매출이 **아예 안 쌓임** → 영원히 매칭 불가
   - 현재 미매칭의 대부분이 이 누락 때문임. 특히 **브랜드커넥트**가 취약(점검 필요)
2. **작업자 닉네임을 성과DB와 똑같이 기입** (띄어쓰기 X)
   - 완료시트의 닉네임과 성과DB(nt_detail)의 닉네임이 글자까지 같아야 매칭됨
3. **제품명 일관되게 기입** (띄어쓰기 X)
4. **날짜 형식 통일** (예: `2026. 7. 8`)

> 💡 매칭이 안 되면 그 작업은 "매출 0"으로 잡혀서 효율이 실제보다 나쁘게 보인다.
> **헤더의 "매칭률 %"이 곧 데이터 건강도 신호** — 낮으면 위 규칙이 안 지켜지고 있다는 뜻.

## 3. 실행 방법

### A. 내 PC에서 혼자 보기
- `run_dashboard.bat` 더블클릭 → 브라우저에서 `http://127.0.0.1:8501` 자동 열림
- 또는 터미널에서: `streamlit run app.py`

### B. 같은 사무실(LAN)에서 여러 명이 보기
- `share_dashboard.bat` 더블클릭 → 표시되는 `http://내IP:8501` 주소를 다른 PC에서 접속

### C. 처음 1회 준비물
- Python 3.11 설치 (`runtime.txt` 기준)
- 의존성 설치: `pip install -r requirements.txt` (배치파일이 자동으로 해줌)

- 대시보드 접속 시 **비밀번호 입력창**이 먼저 뜸
- 현재 비밀번호: **`atwo1926`**
- 변경: `.streamlit/secrets.toml` 파일의 `app_password` 값 수정
- ⚠️ 이 파일은 깃허브에 **안 올라감**(보안). 다른 PC/배포처에는 따로 만들어줘야 함 (4번 참고)

## 5. 매칭이 어떻게 동작하나 (이해용)

- **매칭 키** = `플랫폼(nt_source)` + `작업자(닉네임)` + `키워드(제품)`
- 비용쪽 한 줄과 매출쪽 한 줄이 이 3개가 같으면 "짝"으로 연결 → ROAS 계산
- 코드가 자동으로 흡수해주는 표기 차이:
  - 띄어쓰기 (`gk5 pro` = `gk5pro`)
  - 한글/영어 (`본네온` = `boneneon`, `본플렉스` = `boneflex`)
  - 날짜 접두어 (`20260224_gm3prote1` → `gm3prote1`)
- **날짜 없는 성과행은 일부러 매칭 안 함** (작업 시점 특정 불가 → 잘못 연결 방지. 매출의 2.4%뿐이라 영향 미미)

## 6. 알려진 한계 (솔직한 현황)

- **현재 매칭률 약 23%.** 원인은 코드가 아니라 **성과DB에 데이터가 비어 있음**:
  - 미매칭 건의 83%가 성과DB에 작업자·제품 데이터 자체가 없음(= NT 추적 누락)
  - 브랜드커넥트: 완료작업 10건인데 성과DB 2행 → 매칭 0%
- 코드 정규화는 할 만큼 했음. **앞으로 2번 규칙(특히 NT 파라미터)을 지켜 데이터를 누적하면 매칭률이 자연히 오름.**

## 7. 자주 겪는 문제 (트러블슈팅)

| 증상 | 원인 / 해결 |
|---|---|
| 비밀번호 입력해도 안 열림 | `.streamlit/secrets.toml`의 `app_password` 확인. 배포처면 Secrets 설정 확인 |
| 숫자가 옛날 그대로 | 사이드바 "원본 다시 불러오기" 클릭 (또는 5분 대기) |
| "일부 시트 로딩 실패" 경고 | 시트 공유 설정/네트워크 확인. gid 변경됐는지 확인 |
| 특정 작업이 효율 0으로 나옴 | 매칭 실패 → 2번 입력 규칙(닉네임/NT) 점검. "매칭 실패 리스트" 표에서 사유 확인 |
| 매칭률이 갑자기 떨어짐 | 최근 입력 데이터가 규칙을 안 지켰을 가능성 → 입력 점검 |

## 8. 향후 개선 여지 (시간 날 때)

- 작업자/플랫폼별 **효율 랭킹 표**(ROAS·전환당비용 정렬) 추가 → "누구를 계속 쓸까" 의사결정용
- 성과DB **기여 결제금액(attributed)** 지표 화면에 노출
- 시트 ID를 `secrets`로 분리(공개 저장소 노출 줄이기)
